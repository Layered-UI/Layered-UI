# Catalog Surface

The **Catalog Surface** is the primary discovery experience within Layered UI. It provides developers with an interactive catalog of reusable UI blocks, allowing them to browse categories, preview components, inspect source code, and install blocks directly into their own projects.

Unlike the marketing surface, which focuses on communicating the project's value, the catalog is a developer-focused experience built around exploration, evaluation, and installation.

The catalog is entirely metadata-driven, making it easy to expand without modifying routing logic or application architecture.

---

# Purpose

The catalog exists to help developers:

* Discover reusable UI blocks.
* Browse components by category.
* Preview components before installation.
* Inspect implementation details.
* Copy installation commands.
* Access related examples.
* Install registry components.

Every feature within the catalog supports one or more of these workflows.

---

# Architecture

The catalog is composed of several independent layers.

```text
Component Catalog
        │
        ├── Category Pages
        ├── Block Metadata
        ├── Block Preview
        ├── Source Viewer
        ├── Installation Commands
        └── Registry Integration
```

Each layer has a single responsibility while working together to create a seamless browsing experience.

---

# Routing

Each category is exposed through a static route generated by the Next.js App Router.

```text
/category
```

Examples include:

```text
/buttons
/charts
/forms
/cards
/navigation
```

Categories are generated from catalog metadata rather than manually maintained routes.

---

# Static Generation

Category pages are statically generated during the build process.

The application:

1. Reads category definitions.
2. Generates route parameters.
3. Builds category pages.
4. Pre-renders previews.
5. Produces optimized static output.

This approach keeps browsing fast while reducing runtime complexity.

---

# Metadata-Driven Design

The catalog is powered by a centralized metadata source.

```text
data/
└── blocks.ts
```

Rather than scanning directories dynamically, every visible block is explicitly declared within the catalog metadata.

A typical block entry includes:

* slug
* title
* category
* preview route
* source file
* related files
* installation metadata

This explicit approach keeps the catalog predictable and easy to maintain.

---

# Block Lifecycle

Every block follows the same lifecycle.


```mermaid
flowchart LR
      A[Preview Component] --> B[Catalog Metadata]
      B --> C[Category Page]
      C --> D[Block Preview]
      D --> E[Registry Package]
```


Because every stage references the same implementation, documentation and installation remain synchronized.

---

# Block Preview

Each catalog item renders through a shared preview component.

The preview infrastructure is responsible for:

* displaying live previews,
* switching between preview and code,
* responsive viewport controls,
* iframe management,
* installation commands,
* copy actions,
* source rendering.

Keeping these features centralized ensures a consistent experience across every block.

---

# Preview Loading

To improve performance, previews are loaded lazily.

```text
Catalog Page
      │
      ▼
Intersection Observer
      │
      ▼
Load Preview
      │
      ▼
Measure Height
      │
      ▼
Cache Layout
```

This strategy reduces unnecessary rendering while improving scrolling performance on pages containing many components.

---

# Responsive Preview

Developers can evaluate components across multiple viewport sizes.

Typical viewport options include:

* Mobile
* Tablet
* Desktop

Changing viewport size affects only the preview iframe rather than the surrounding catalog page.

---

# Source Code

Each block exposes its implementation directly within the catalog.

Developers can:

* inspect the implementation,
* copy code,
* compare variants,
* understand dependencies.

Syntax highlighting is handled through the shared code rendering infrastructure.

---

# Installation Workflow

The catalog generates installation commands automatically.

```text
Browse Block
      │
      ▼
Copy Install Command
      │
      ▼
Run CLI
      │
      ▼
Install Component
```

Installation commands are derived from the same metadata that powers the catalog, reducing duplication and configuration errors.

---

# Registry Integration

The catalog does not directly install components.

Instead, it references registry definitions that describe:

* dependencies,
* target files,
* registry items,
* package requirements.

This separation keeps browsing concerns independent from distribution concerns.

---

# Performance Considerations

Several optimizations keep the catalog responsive.

These include:

* Static category generation
* Lazy iframe loading
* Cached iframe heights
* Shared preview infrastructure
* Efficient metadata lookup
* Code splitting
* Lightweight client components

Together, these strategies allow the catalog to scale without significantly increasing page load times.

---

# Extending the Catalog

Adding a new block follows a predictable workflow.

1. Create a preview page.
2. Add a metadata entry to `data/blocks.ts`.
3. Register the block in `registry.json`.
4. Generate registry assets.
5. Verify the preview and installation workflow.

No routing changes are required.

---

# Design Principles

The catalog is guided by several architectural principles.

## Metadata Over Discovery

Catalog entries should be declared explicitly rather than inferred from the filesystem.

---

## One Preview, Many Uses

Every preview should simultaneously support:

* documentation,
* browsing,
* source viewing,
* registry generation.

---

## Fast by Default

The browsing experience should remain responsive regardless of catalog size.

---

## Consistent User Experience

Every block should expose the same interactions, including preview, source code, installation, and responsive testing.

---

# Summary

The Catalog Surface is the central developer experience of Layered UI.

It transforms metadata and preview components into a searchable, interactive component library that allows developers to evaluate, understand, and install reusable UI blocks. By combining static generation, reusable preview infrastructure, and registry integration, the catalog remains scalable, maintainable, and performant as the project grows.
